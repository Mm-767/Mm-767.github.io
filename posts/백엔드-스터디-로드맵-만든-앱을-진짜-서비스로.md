> 목표: 지금까지 만든 Spring Boot + JPA 앱(Member·Assignment CRUD)을 **문서화 → 견고화 → 보안 → 배포**까지 끌고 가 완성된 백엔드 서비스로 만든다. 대상: CRUD·JPA·연관관계·트랜잭션까지 끝낸 백엔드 스터디원 형식: 매 주차 = **PPT + 스터디 진행 대본 + 예제 코드** 세트 총 5주 (테스트 주차는 선택)

* * *

## 한눈에 보기

| 주차 | 주제 | 한 줄 목표 | 산출물 | 소요 |
| --- | --- | --- | --- | --- |
| 1 | Swagger + 예외 처리·검증 | 흩어진 null 체크를 전역 예외로, 요청은 @Valid로 | 전역 예외 핸들러 · 공통 응답 · Swagger UI | 4h |
| 2 | 인증/인가 (JWT) | 로그인하면 토큰, 토큰으로 API 보호 | 회원가입·로그인·JWT 필터·인가 | 6h |
| 3 | 테스트 (선택·짧게) | API가 진짜 도는지 자동으로 검증 | 통합 테스트 · MockMvc | 3h |
| 4 | 도커 | 앱 + MySQL을 컨테이너 한 번에 | Dockerfile · docker-compose | 4h |
| 5 | 배포 (AWS) | 인터넷에 API 서버 띄우기 | EC2 배포 · Swagger 공개 링크 | 5h |

> **왜 이 순서인가**: 각 주차가 "이전 주차의 약점"을 메운다. 지금 코드의 흩어진 null 체크 → (1) 전역 예외. 누구나 API 호출 가능 → (2) 인증. 손으로만 검증 → (3) 테스트. 내 노트북에서만 돎 → (4)(5) 컨테이너·배포.

* * *

## 1주차 — Swagger + 예외 처리·검증

**한 줄 목표**: API 문서를 자동 생성하고, 에러 처리를 한곳으로 모은다.

**왜 지금**: 방금 끝낸 미션의 `null 반환 + 컨트롤러 null 체크`는 컨트롤러마다 코드가 흩어진다. 이걸 **전역 예외 처리로 리팩터링**하는 게 자연스러운 다음 단계. Swagger를 먼저 깔면 이후 모든 주차를 브라우저로 시연할 수 있다.

**다룰 것**

*   `springdoc-openapi` 의존성 → `/swagger-ui.html`에서 API 문서·테스트
    
*   `@RestControllerAdvice` + `@ExceptionHandler`로 전역 예외 처리
    
*   커스텀 예외(`MemberNotFoundException` 등) + 공통 에러 응답 포맷(`ErrorResponse`)
    
*   `@Valid` + `@NotBlank`/`@Size`로 요청 DTO 검증, `MethodArgumentNotValidException` 처리
    

**산출물**: Swagger 연동, `GlobalExceptionHandler`, 공통 응답 클래스, 검증 붙은 DTO

**끝나면 이렇게 달라진다**: 컨트롤러에서 `if (x == null) return 404` 가 사라지고, 서비스는 예외만 던지면 된다. 잘못된 요청은 자동으로 400 + 친절한 메시지.

* * *

## 2주차 — 인증/인가 (Spring Security + JWT)

**한 줄 목표**: 로그인한 사람만 API를 쓰게 만든다.

**왜 지금**: 지금은 누구나 모든 API를 호출할 수 있다. 실무·포트폴리오·면접에서 **가장 임팩트 큰 주제**가 인증이다.

**다룰 것**

*   Spring Security 기본 흐름(필터 체인), 비밀번호 암호화(`BCrypt`)
    
*   회원가입 / 로그인 API → 로그인 성공 시 **JWT 발급**
    
*   `JwtAuthenticationFilter`로 요청마다 토큰 검증
    
*   인가: 특정 API는 로그인 필요, 본인 과제만 수정 등
    
*   `SecurityContext`에서 현재 사용자 꺼내 쓰기
    

**산출물**: 로그인/회원가입, JWT 발급·검증 필터, 보호된 API

**주의**: Security는 처음에 "왜 401이 뜨지"로 헤매기 쉽다. 대본에 **디버깅 포인트**(필터 순서, 토큰 헤더 형식)를 꼭 넣을 것.

* * *

## 3주차 — 테스트 (선택 · 짧게)

**한 줄 목표**: 손으로 Postman 누르는 대신, 코드가 API를 검증하게.

**왜 선택**: 중요하지만 부원 체력에 따라 건너뛰거나 30분 미니세션으로 가능. 2주차 인증까지 붙은 API가 있어 검증할 거리가 풍부해진 지금이 적기.

**다룰 것**

*   `@SpringBootTest` + `MockMvc`로 API 통합 테스트
    
*   성공/실패(404·400·401) 케이스 검증
    
*   트랜잭션 롤백으로 테스트 격리 (앞서 배운 개념 재활용)
    

**산출물**: 핵심 API 통합 테스트 몇 개

* * *

## 4주차 — 도커

**한 줄 목표**: "내 컴퓨터에선 되는데요"를 없앤다.

**왜 지금**: 배포 전에 앱과 DB를 컨테이너로 묶어두면, 5주차 배포가 훨씬 쉬워진다.

**다룰 것**

*   이미지 vs 컨테이너 개념, `Dockerfile`로 Spring Boot 앱 이미지 만들기
    
*   `docker-compose`로 **앱 + MySQL 한 번에** 실행
    
*   환경변수로 DB 접속 정보 분리(`application.yml` 프로파일)
    

**산출물**: `Dockerfile`, `docker-compose.yml`, `docker compose up` 한 방 실행

**성취 포인트**: 명령어 하나로 앱+DB가 뜨는 걸 보면 부원들 반응이 제일 좋은 주차.

* * *

## 5주차 — 배포 (AWS)

**한 줄 목표**: 인터넷에 우리 API를 띄우고, Swagger 링크로 시연한다.

**다룰 것**

*   AWS EC2 인스턴스 생성, 보안 그룹(포트) 설정
    
*   Docker 설치 → 만든 이미지 실행 (또는 compose)
    
*   DB: RDS 사용 vs 컨테이너 MySQL 트레이드오프
    
*   `배포주소/swagger-ui.html` 공개 → **프론트 없이도 시연 완료**
    

**산출물**: 외부에서 접속되는 API 서버, 공유 가능한 Swagger 링크

**마무리 활동**: 각자 배포 주소를 공유하고 서로의 API를 Postman으로 호출해보기.

* * *

## 운영 팁

*   **매주 "지난주 코드의 약점 → 이번주 해결"** 을 첫 슬라이드에 넣으면 동기부여가 된다.
    
*   **개념 30% + 실습 70%.** 특히 Security·Docker는 눈으로 돌아가는 걸 봐야 이해된다.
    
*   각 주차 끝에 **작은 과제**(다음 주까지 각자 적용해오기)를 낸다.
    
*   부원 수준 편차가 크면 **테스트(3주차)는 미니세션**으로 줄이고 그 시간을 Security 복습에 쓴다.
