> "모델이 아니라 \*\*모델을 감싼 하네스(harness)\*\*가 실력을 가른다." Claude Code를 예시로, 바이브 코딩을 프로덕션 수준으로 끌어올리는 하네스 엔지니어링 전부. (참고: 최신 기능 기준. Claude Code의 훅·플러그인은 계속 진화하므로 공식 문서로 교차확인 권장)

* * *

## 0\. 용어

바이브 코딩(vibe coding)이라는 말을 만든 사람은 **Andrej Karpathy(안드레이 카파시)** 입니다. 테슬라 AI 총괄, OpenAI 창립 멤버 출신이고 "Software 1.0/2.0/3.0" 프레임으로 유명합니다.

* * *

## 1\. 하네스 엔지니어링이란?

**하네스(harness)** 는 LLM 자체가 아니라, **그 모델을 둘러싼 실행 환경 전체**를 말합니다. 컨텍스트를 채우는 방식, 쓸 수 있는 도구, 자동으로 끼어드는 훅, 일을 나눠 맡는 서브에이전트, 권한 규칙 — 이 스캐폴딩의 총합이 하네스입니다.

핵심 통찰은 이거예요. **요즘 성능 차이는 모델보다 하네스에서 난다.** 같은 모델이라도 컨텍스트를 잘 관리하고, 검증 루프를 붙이고, 도구를 잘 쥐여주면 결과물 품질이 완전히 달라집니다. 그래서 "프롬프트 엔지니어링"을 넘어 **하네스 엔지니어링 / 컨텍스트 엔지니어링**이 핵심 역량으로 부상했습니다.

### 카파시의 두 축 — vibe coding vs agentic engineering

카파시는 AI 코딩을 두 가지로 선을 긋습니다.

*   **Vibe coding (바이브 코딩)** — "이거 만들어줘" 하면 에이전트가 코드를 뱉고, 대충 맞으면 받아들이고 넘어가는 방식. **바닥(floor)을 올린다** — 누구나 빠르게 프로토타입을 만들 수 있게 함. 빠르고 접근성 좋지만 프로덕션엔 위험.
    
*   **Agentic engineering (에이전틱 엔지니어링)** — 강력하지만 불안정한 에이전트를 **규율 있게 조율**해서 보안·품질을 갖춘 소프트웨어를 만드는 것. **천장(ceiling)을 올린다.**
    

> 하네스 엔지니어링은 바이브 코딩을 에이전틱 엔지니어링으로 끌어올리는 기술이라고 보면 됩니다.

### 생성-검증 루프 (Generation–Verification Loop)

카파시가 수년간 강조한 핵심 원칙:

> **"AI를 목줄에 묶어두고(keep AI on a leash), 생성-검증 루프의 속도를 최대화하라."**

에이전트가 코드를 **생성(generate)** 하면, 사람이나 자동 검사가 **검증(verify)** 한 뒤 다음 단계로 넘어가는 사이클입니다. 프롬프트 → 생성 → 검증 → 재프롬프트. 하네스 엔지니어링의 목표는 이 **검증 단계를 빠르고 자동적으로** 만드는 것 — 테스트, 린트, 타입체크, 스크린샷 확인 등을 루프에 끼워 넣어 사람의 확인 부담을 줄이는 거예요.

### Software 3.0와 컨텍스트 엔지니어링

카파시의 프레임:

*   **Software 1.0** — 사람이 명시적으로 짠 코드
    
*   **Software 2.0** — 데이터로 학습된 신경망 가중치
    
*   **Software 3.0** — **프롬프트가 곧 프로그래밍.** LLM을 "디지털 정보 처리 인터프리터"로 보고, **컨텍스트 윈도우가 그 LLM을 조종하는 레버**가 됨.
    

즉 Software 3.0 시대의 프로그래밍은 **컨텍스트를 설계하는 일** — 이게 컨텍스트 엔지니어링입니다.

* * *

## 2\. 컨텍스트 엔지니어링 — 가장 근본 기술

LLM은 **컨텍스트 윈도우** 안에 있는 것만 볼 수 있습니다. 이 윈도우는 유한한 자원이고, 다음 세 가지가 품질을 결정해요.

1.  **무엇을 넣을까** — 지금 작업에 꼭 필요한 파일·규칙·예시만. 관련 없는 건 노이즈.
    
2.  **무엇을 뺄까** — 오래된/무관한 내용이 쌓이면 "컨텍스트 오염"이 생겨 모델이 헷갈림.
    
3.  **언제 압축할까** — 긴 세션에선 요약(compaction)으로 핵심만 남기고 토큰을 회수.
    

**실전 원칙**

*   한 세션 = 한 목표. 주제가 바뀌면 새 세션(또는 `/clear`)으로 컨텍스트를 리셋.
    
*   큰 탐색 작업은 **서브에이전트**에 맡겨서 결과 요약만 메인 컨텍스트로 가져오기(오염 방지).
    
*   반복되는 프로젝트 규칙은 대화에 매번 붙이지 말고 **CLAUDE.md**에 고정.
    

* * *

## 3\. Claude Code라는 하네스 — 구성 요소 지도

Claude Code는 잘 만들어진 코딩 하네스입니다. 핵심 부품을 지도로 보면:

| 부품 | 역할 | 한 줄 |
| --- | --- | --- |
| **CLAUDE.md** | 프로젝트 메모리 | 항상 로드되는 규칙·컨벤션 |
| **Tools** | 실행 능력 | 파일 읽기/쓰기, Bash, 검색 등 |
| **Permissions** | 안전장치 | 무엇을 자동 허용/차단할지 |
| **MCP 서버** | 외부 연결 | Slack·GitHub·DB 등 도구 확장 |
| **Subagents** | 컨텍스트 분리 | 별도 창에서 하위 작업 수행 |
| **Slash commands** | 재사용 프롬프트 | `/명령`으로 워크플로우 호출 |
| **Skills** | 절차적 전문성 | 특정 작업 방법을 담은 패키지 |
| **Hooks** | 결정론적 자동화 | 라이프사이클에 끼어드는 스크립트 |
| **Plugins** | 번들 배포 | 위 요소들을 한 번에 설치 |

아래에서 하나씩 봅니다.

* * *

## 4\. CLAUDE.md — 프로젝트 메모리

프로젝트 루트(또는 `~/.claude/CLAUDE.md`, 서브디렉터리)에 두면 **세션 시작 시 자동으로 컨텍스트에 로드**됩니다. 매번 반복 설명할 규칙을 여기에 고정하는 게 컨텍스트 엔지니어링의 1번 실천법.

담을 것 예시:

*   빌드/테스트 명령어 (`./gradlew test`)
    
*   코드 컨벤션 (네이밍, 포맷, 금지 패턴)
    
*   아키텍처 요약 (레이어 구조, 폴더 규칙)
    
*   "하지 말 것" (예: `main`에 직접 커밋 금지)
    

```markdown
# 프로젝트 규칙
- 테스트: `./gradlew test` / 린트: `./gradlew ktlintCheck`
- 예외는 GlobalExceptionHandler로만 처리, 컨트롤러에서 null 반환 금지
- 커밋 전 항상 테스트 통과 확인
```

> 팁: CLAUDE.md는 **짧고 규칙 중심**으로. 장황하면 그 자체가 컨텍스트를 잡아먹습니다.

* * *

## 5\. 도구(Tools)와 권한(Permissions)

에이전트의 "손"이 도구입니다. 파일 Read/Write/Edit, Bash 실행, 검색(Grep/Glob), 웹 등. **하네스 설계 = 어떤 도구를, 어떤 권한으로 쥐여줄지 정하는 일**이기도 합니다.

권한 모드 감각:

*   위험한 작업(삭제, 네트워크, 결제)은 **확인 후 실행**.
    
*   안전한 반복 작업(읽기, 포맷)은 **자동 허용**해서 루프 속도 ↑.
    
*   권한을 너무 조이면 느리고, 너무 풀면 위험 — **생성-검증 루프 속도와 안전의 균형**이 핵심.
    

* * *

## 6\. MCP 서버 — 외부 세계 연결

**MCP(Model Context Protocol)** 는 Claude Code에 외부 도구를 붙이는 표준입니다. GitHub, Slack, Postgres, Sentry, 브라우저 등 MCP 서버를 연결하면 그 도구들이 에이전트의 도구 목록에 추가돼요.

*   설정: 프로젝트의 `.mcp.json` 또는 `claude mcp` 명령.
    
*   예: DB MCP를 붙이면 에이전트가 직접 스키마를 조회하고 쿼리를 검증.
    
*   하네스 관점: MCP는 **에이전트의 감각·행동 범위를 넓히는** 확장 슬롯.
    

* * *

## 7\. 서브에이전트(Subagents) — 컨텍스트를 나눈다

서브에이전트는 **자기만의 컨텍스트 윈도우, 시스템 프롬프트, 도구 세트를 가진 별도 AI**입니다. 메인 에이전트가 특정 작업을 만나면 서브에이전트에게 위임하고, 서브에이전트는 독립적으로 일한 뒤 **결과만 요약해서 반환**합니다.

**왜 중요한가 (컨텍스트 엔지니어링의 꽃)**

*   대규모 코드 탐색, 로그 분석 같은 "토큰 많이 먹는" 작업을 서브에이전트에 맡기면 **메인 컨텍스트가 오염되지 않음.**
    
*   검증 전용 서브에이전트(예: "이 diff를 리뷰해") 를 두면 생성-검증 루프를 자동화.
    
*   병렬로 여러 서브에이전트를 돌려 팬아웃 탐색.
    

**정의 위치**: `.claude/agents/*.md` (프론트매터로 이름·설명·모델·도구 지정). 플러그인으로도 배포됩니다.

* * *

## 8\. 슬래시 커맨드(Slash Commands)

`/명령` 형태로 호출하는 **재사용 가능한 프롬프트/워크플로우**입니다. 자주 하는 작업(리뷰, 릴리스 노트 작성, PR 요약)을 한 단어로 실행.

*   정의: `.claude/commands/*.md` — 파일 하나가 커맨드 하나.
    
*   인자 전달, 파일 참조 등 지원.
    
*   플러그인에 포함해 팀 전체가 공유 가능.
    

* * *

## 9\. 스킬(Skills)

스킬은 **특정 종류의 작업을 어떻게 처리하는지** 담은 패키지입니다(SKILL.md + 보조 스크립트). "무엇을 아느냐"가 아니라 "어떻게 하느냐"의 절차적 전문성.

*   예: docx/pptx/pdf 생성, 배포 체크리스트, 리뷰 절차.
    
*   에이전트는 작업이 스킬과 맞으면 자동으로 그 절차를 불러와 따름.
    
*   슬래시 커맨드와 차이: 커맨드는 "내가 호출", 스킬은 "필요할 때 자동 발동"에 가깝습니다.
    

* * *

## 10\. 훅(Hooks) — 결정론적 자동화 레이어 ⭐

훅은 하네스 엔지니어링의 **핵심 무기**입니다. LLM은 확률적이라 "항상 이렇게 해"를 100% 보장 못 하는데, 훅은 **라이프사이클의 특정 시점에 무조건 실행되는 스크립트**라서 **결정론적 규칙**을 강제할 수 있어요. (린트 자동 실행, 위험 명령 차단, 컨텍스트 자동 주입 등)

### 어떻게 동작하나

이벤트가 발생하고 **matcher(필터)** 가 맞으면, Claude Code가 이벤트에 대한 JSON을 훅 핸들러에 넘깁니다(커맨드 훅은 stdin으로, HTTP 훅은 POST 바디로). 핸들러는 입력을 검사하고 행동한 뒤, 선택적으로 **결정(decision)** 을 반환할 수 있어요.

*   설정 위치: `~/.claude/settings.json`(전역) 또는 프로젝트 `.claude/settings.json`
    
*   핸들러 종류: 셸 커맨드 / HTTP 엔드포인트 / MCP 도구 / 프롬프트 / 에이전트
    
*   **Exit code 규약**:
    
    *   `0` = 성공. stdout의 JSON을 파싱해 결정 필드 처리.
        
    *   `2` = **블로킹 에러**. stdout 무시, **stderr 텍스트를 에러로 모델에 되먹임** → 이걸로 툴 호출을 막거나 재작업 유도.
        

### 대표 훅 이벤트 (공식)

| 이벤트 | 발동 시점 | 주요 용도 |
| --- | --- | --- |
| `SessionStart` | 세션 시작/재개 | 프로젝트 메트릭·티켓 등 **컨텍스트 자동 주입** |
| `UserPromptSubmit` | 프롬프트 제출 직후(처리 전) | 프롬프트 검사·보강, 차단 |
| `PreToolUse` | 도구 호출 **직전** | **승인/거부/수정** — 가장 강력. 위험 명령 차단 |
| `PostToolUse` | 도구 호출 성공 후 | 결과 검사, 자동 린트/포맷, 피드백 |
| `PostToolUseFailure` | 도구 호출 실패 후 | 실패 로깅·복구 힌트 |
| `SubagentStart` / `SubagentStop` | 서브에이전트 시작/종료 | 위임 추적 |
| `Stop` | Claude가 응답을 마칠 때 | 완료 후 검증·알림 |
| `Notification` | 알림 발생 시 | 데스크톱 핑 등 |
| `PreCompact` / `PostCompact` | 컨텍스트 압축 전/후 | 압축 전 핵심 보존 |
| `FileChanged` | 감시 파일 변경 시 | 반응형 재빌드 |
| `SessionEnd` | 세션 종료 | 정리·요약 저장 |

> 이 외에도 `Setup`, `PermissionRequest`, `PermissionDenied`, `PostToolBatch`, `TaskCreated/Completed`, `CwdChanged`, `WorktreeCreate/Remove`, `Elicitation` 등 세분화된 이벤트가 있습니다(총 30개 안팎). 세션당 1회(`SessionStart`/`SessionEnd`), 매 도구 호출마다(`PreToolUse`/`PostToolUse`) 같은 식으로 발동 빈도가 다릅니다.

### matcher (필터)

`matcher` 필드로 "언제 발동할지"를 좁힙니다.

*   `PreToolUse`/`PostToolUse` 등은 **도구 이름**으로 매칭: `Bash`, `Edit|Write`, `mcp__.*`
    
*   정규식 경로로 평가(`RegExp.test`) — `Edit.*`는 `Edit`과 `NotebookEdit` 둘 다 매치. 정확히 맞추려면 `^Edit$`.
    
*   생략하거나 `"*"` 이면 해당 이벤트마다 항상 발동.
    

### 예제 1 — 위험한 `rm` 차단 (PreToolUse)

`.claude/settings.json`:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          { "type": "command", "command": "block-rm.sh" }
        ]
      }
    ]
  }
}
```

`block-rm.sh` — `rm -rf` 류면 exit 2로 막고 stderr로 이유 전달:

```bash
#!/usr/bin/env bash
input=$(cat)                       # stdin으로 JSON 입력
cmd=$(echo "$input" | jq -r '.tool_input.command')
if echo "$cmd" | grep -qE 'rm\s+-rf'; then
  echo "위험한 rm -rf 명령은 차단됩니다: $cmd" >&2   # stderr → 모델에 피드백
  exit 2                                              # 블로킹
fi
exit 0
```

### 예제 2 — 저장할 때마다 자동 포맷 (PostToolUse)

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          { "type": "command", "command": "npx prettier --write \"$CLAUDE_FILE_PATH\"" }
        ]
      }
    ]
  }
}
```

### 예제 3 — 세션 시작 시 컨텍스트 주입 (SessionStart)

```json
{
  "hooks": {
    "SessionStart": [
      { "hooks": [ { "type": "command", "command": "git log --oneline -5 && cat TODO.md" } ] }
    ]
  }
}
```

> 훅의 진짜 가치: **"에이전트가 알아서 잘하겠지"에 의존하지 않고, 중요한 규칙을 코드로 강제**한다는 것. 생성-검증 루프의 "검증"을 자동화하는 곳이 바로 훅입니다.

* * *

## 11\. 플러그인 & 마켓플레이스

**플러그인**은 위에서 본 요소들 — 슬래시 커맨드, 서브에이전트, 훅, MCP 서버, 스킬 — 을 **하나의 설치 단위로 번들**한 것입니다. (2025년 10월 9일 출시)

### 구조

```text
my-plugin/
├── .claude-plugin/          # 플러그인 메타데이터
│   └── plugin.json
├── .mcp.json                # 번들 MCP 서버 설정
├── commands/                # 슬래시 커맨드
├── agents/                  # 서브에이전트
├── skills/                  # 스킬
├── hooks/                   # 훅
└── README.md
```

### 설치·배포

*   설치: `/plugin install <이름>@<마켓플레이스>`
    
*   **마켓플레이스** = 플러그인 묶음을 배포하는 저장소(공식 + 서드파티/커뮤니티).
    
*   팀 표준(린트 훅 + 리뷰 커맨드 + 배포 스킬)을 플러그인 하나로 묶어 조직 전체에 배포 가능.
    

> 하네스 관점: 플러그인은 **하네스 설정을 재사용·공유 가능한 자산으로** 만드는 장치. 잘 만든 하네스를 팀·커뮤니티에 그대로 나눠줄 수 있습니다.

## 12\. 실전 워크플로우 — 하네스를 실제로 굴리는 법

### A. 계획 먼저 (Plan Mode)

큰 작업은 바로 코딩시키지 말고, **먼저 계획을 세우게** 하고 사람이 승인한 뒤 실행. 생성-검증 루프의 "검증"을 코딩 전으로 당기는 것.

### B. 테스트 주도 (검증 자동화)

*   기능 전에 테스트를 만들거나, 작업 끝마다 테스트를 돌리는 훅(`Stop`/`PostToolUse`)을 건다.
    
*   실패하면 훅이 exit 2로 되먹여 에이전트가 스스로 고치게.
    

### C. 서브에이전트로 검증 분리

*   구현은 메인, **리뷰·검증은 별도 서브에이전트**에 위임 → 편향 없는 점검 + 컨텍스트 절약.
    

### D. 컨텍스트 위생

*   한 세션 한 목표, 주제 바뀌면 리셋.
    
*   반복 규칙은 CLAUDE.md로, 대규모 탐색은 서브에이전트로.
    

### E. 목줄을 짧게, 루프를 빠르게

*   카파시 원칙 그대로: 한 번에 거대한 변경을 시키지 말고 **작게 생성 → 즉시 검증 → 반복**. 검증이 빠를수록 목줄을 길게 풀어도 안전.
    

* * *

## 13\. 안티패턴 (하지 말 것)

*   **컨텍스트에 다 때려넣기** — 관련 없는 파일까지 붙이면 오히려 품질 저하.
    
*   **검증 없이 수용** — "돌아가는 것처럼 보임"으로 병합. 프로덕션 사고의 지름길.
    
*   **훅 없이 규칙을 말로만** — "항상 테스트 돌려"는 확률적. 훅으로 강제하라.
    
*   **거대한 단일 프롬프트** — 목줄이 너무 길면 방향을 잃음. 작게 쪼개라.
    
*   **권한 전부 열기** — 속도는 빨라도 위험. 위험 작업은 확인 유지.
    

* * *

## 14\. 시작 체크리스트

1.  프로젝트 루트에 **CLAUDE.md** 작성 (빌드·테스트·컨벤션·금지사항)
    
2.  자주 쓰는 작업을 **슬래시 커맨드**로 정리 (`/review`, `/release-notes`)
    
3.  필요한 **MCP 서버** 연결 (GitHub·DB 등)
    
4.  **PreToolUse 훅**으로 위험 명령 차단, **PostToolUse 훅**으로 자동 포맷/린트
    
5.  **테스트 자동 실행 훅**으로 생성-검증 루프 자동화
    
6.  큰 탐색/검증은 **서브에이전트**로 분리
    
7.  팀 표준이 잡히면 **플러그인**으로 묶어 배포
    
8.  항상 **plan → 생성 → 검증 → 반복**, 목줄은 짧게

## 한 문장 요약

> **하네스 엔지니어링 = 컨텍스트를 설계하고(CLAUDE.md·서브에이전트), 능력을 쥐여주고(도구·MCP), 규칙을 강제하고(훅·권한), 이걸 재사용 가능하게 묶어서(플러그인) — 카파시의 생성-검증 루프를 빠르고 안전하게 돌리는 기술.**
