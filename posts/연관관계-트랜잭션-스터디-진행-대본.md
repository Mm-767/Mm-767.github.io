**대본**

## 3장 — 시나리오 (1:N)

**대본**

> 규칙 두 개. **회원 1명이 과제 여러 개**를 갖고, **과제 하나는 회원 1명**에게 속해요. 회사-직원 관계랑 똑같죠. 이걸 객체랑 테이블에 어떻게 담느냐가 오늘 핵심이에요.

**❓던질 질문**: "이런 1:N 관계, 여러분 프로젝트에서 뭐가 있었어요?" → 사례 하나씩

* * *

## 4장 — ① 연관관계의 주인 = N쪽(FK)

**대본**

> 제일 중요한 개념. **객체는 참조가 2개**(회원↔과제), **DB는 FK 하나**. 개수가 안 맞으니 누가 FK를 관리할지 '주인'을 정해요. 규칙은 **FK를 가진 N쪽, Assignment가 주인**.
> 
> 왜냐? **FK인 member\_id가 assignment 테이블에 생기거든요.** 그 컬럼 가진 쪽이 관계를 책임지는 게 자연스럽죠.

**❓던질 질문**: (설명 전에) "회원이랑 과제 중 누가 주인일 것 같아요? 왜요?" → 답 듣고 정답 공개

**⚠️ 실수 짚기**: "mappedBy는 주인이 아니라 \*\*주인이 아닌 쪽(Member)\*\*에 붙어요!"

* * *

## 5장 — ② mappedBy · @JoinColumn · FK 위치

**대본**

> 두 어노테이션의 역할을 확실히 나눠요.
> 
> *   **@JoinColumn** → Assignment(주인)에. `assignment` 테이블에 `member_id` FK를 만든다.
>     
> *   **mappedBy** → Member(조회용)에. "FK는 Assignment가 관리한다"는 선언. 읽기 전용.
>     
> *   그래서 **member 테이블엔 아무 컬럼도 안 생겨요.** FK는 항상 N쪽에.
>     
> 
> 컬렉션은 `new ArrayList<>()`로 초기화 — NPE 방지.

**❓던질 질문**: "mappedBy를 붙인 Member 리스트에 과제를 add하면 DB에 저장될까요?" → (답: 안 됨, 주인이 아니라서)

## 6장 — ③ 만들 것: Assignment CRUD API

**대본**

> 실제로 만들 API는 5개예요. **등록(POST, 201) / 멤버별 목록(GET) / 단건(GET) / 수정(PUT) / 삭제(DELETE, 204).** 등록만 `/members/{memberId}/assignments`로 memberId를 받는 거 주의하고요.
> 
> 그리고 제약사항 — **Assignment 코드는 전부** `assignment` **패키지**(controller/service/repository/domain/dto)로 분리해요.

**❓던질 질문**: "등록 API만 URL에 memberId가 들어가요. 왜 그럴까요?" → (누구의 과제인지 알아야 하니까)

## 7장 — ④ 트랜잭션을 왜 Service에?

**대본**

> 트랜잭션 경계는 **Service**에 둬요. '과제 등록'이 사실 회원 조회 + 과제 저장 묶음이거든요.
> 
> 이걸 Repository마다 걸면 중간에 터졌을 때 **앞 작업은 이미 커밋돼서 못 돌려요.** **계좌이체**처럼 출금·입금이 같이 성공하거나 같이 실패해야 하는 거죠.

**❓던질 질문**: "Repository마다 트랜잭션 걸면 뭐가 문제일까요?" → 원자성 스스로 도출

* * *

## 8장 — ⑤ 트랜잭션 전략: readOnly = true

**대본**

> 미션의 핵심 전략이에요. **2단으로 구성**해요.
> 
> *   클래스에 `@Transactional(readOnly = true)` → 기본은 전부 읽기 전용.
>     
> *   **등록·수정·삭제 메서드에만** `@Transactional`을 따로 붙여 읽기 전용을 풀어요.
>     
> 
> 왜? **읽기 전용은 변경 감지를 생략해서 조회가 빨라요.** 조회 메서드엔 안 붙이고 클래스 기본을 그대로 쓰는 거죠. 수정은 `updateInfo()`로 필드 바꾸고 `save()`.

**❓던질 질문**: "조회 메서드엔 왜 @Transactional을 안 붙이고 그냥 둘까요?" → readOnly 성능 이점

* * *

## 9장 — ⑥ 에러 처리: null → 404

**대본**

> 이 미션은 **커스텀 예외를 안 써요.** week8 방식 그대로.
> 
> *   **Service**: 대상이 없으면 `null` 반환.
>     
> *   **Controller**: null이면 **404 Not Found**.
>     
> 
> 상태 코드는 등록 201, 조회·수정 200, 삭제 204, 없으면 404. 딱 이거예요.

**❓던질 질문**: "없는 멤버한테 과제를 등록하면 어떤 응답을 줘야 할까요?" → 404

* * *

## 10장 — 핵심 코드

**대본**

> 코드로 정리하면 명확해요. **왼쪽 Assignment**가 `@JoinColumn`으로 FK를 쥔 주인, 필드는 `id·title·description·member`. **오른쪽 Member**는 `mappedBy`로 조회 전용 + `ArrayList` 초기화.
> 
> 다시 강조 — **member 테이블엔 추가 컬럼이 안 생겨요.**

**❓던질 질문**: "Assignment 필드에 뭐가 들어가야 하죠?" → id, title, description, member

* * *

## 11장 — 실행 & 검증

**대본**

> 마지막은 진짜 도는지 확인이에요. 세 가지.
> 
> 1.  **콘솔 SQL** — 시작할 때 `create table assignment`에 `member_id`랑 FK 제약이 찍히는지.
>     
> 2.  **Postman** — 등록→목록→단건→수정→삭제, 없는 멤버 등록은 404.
>     
> 3.  **MySQL** — `assignment`와 `member`를 JOIN해서 올바른 회원 id가 저장됐는지.
>     

**❓던질 질문**: "콘솔에서 뭘 보면 FK가 잘 만들어졌다고 확신할 수 있을까요?" → member\_id 컬럼 + FK 제약

## 12장 — 핵심 요약 3줄

**대본**

> 오늘 딱 세 개만. **하나, 주인은 FK 가진 N쪽. 둘, 트랜잭션은 Service에 readOnly 기본 + 변경만 @Transactional. 셋, 에러는 예외 없이 null → 404.**
> 
> 결국 다 **데이터 안 깨지게 하는 규칙**이에요.
